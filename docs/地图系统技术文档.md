## 二、数据模型

### Exit（出口）

地图的核心数据单元，每个房间有若干个 `Exit`，描述"从这里能去哪"。

```dart
class Exit {
  final String dir;         // 方向代码：n/s/e/w/ne/nw/se/sw/u/d
  final String label;       // 中文标签：北/南/东/西/东北/西北/东南/西南
  final String toRoom;      // 目标房间 ID
  final String toName;      // 目标房间名称（直接用于渲染，无需二次查询）
  final bool crossRegion;   // 是否跨区域（影响连线颜色）
  final String? portal;     // 跨区域通道名（可选）
}
```

### PositionData（位置数据）

每次移动或刷新后服务端返回的完整状态：

```dart
class PositionData {
  final CurrentPosition current;       // 当前房间信息
  final List<Exit> exits;              // 当前房间的所有出口
  final List<AdjacentRegion> adjacentRegions;
  final List<MapNode> mapNodes;        // 区域内所有节点（用于 MapPainter）
  final WorldState world;              // 世界时间/天气等
}
```

**关键设计**：`exits` 里的 `toName` 直接包含目标房间名，地图渲染时不需要再查房间表，减少一次数据关联。

## 三、地图渲染原理

### 3.1 渲染方案选型

| 方案 | 问题 |
|------|------|
| Widget 九宫格布局 | 斜向节点位置固定，连线是视觉假象（无真实线条），斜向连线不自然 |
| CustomPainter 圆形节点 | 圆形节点信息密度低，8方向全显示时空节点干扰视觉 |
| **CustomPainter 矩形节点（当前方案）** | 只渲染有出口的方向，连线从矩形边缘精确出发，斜向自然 |

### 3.2 ExitMapPainter 核心原理

**坐标系**：以画布中心为当前房间，8个方向按固定 `step` 距离放置邻居节点。

```
step = 110.0px

nw(-step,-step)  n(0,-step)  ne(+step,-step)
w(-step, 0)      [当前]      e(+step, 0)
sw(-step,+step)  s(0,+step)  se(+step,+step)
```

屏幕坐标系 y 轴向下，所以"北"对应 `Offset(0, -step)`，"东南"对应 `Offset(+step, +step)`，斜向节点天然在45°方向上，无需三角函数。

**绘制顺序**（painter's algorithm，后画的覆盖先画的）：

```
1. 画所有连线  →  2. 画邻居节点  →  3. 画当前节点（最上层）
```

当前节点最后画，确保连线端点被节点矩形遮住，视觉上线从矩形边缘"长出来"。

### 3.3 连线边缘裁剪算法

连线不能从矩形中心到中心（会穿透矩形），需要计算射线与矩形边的交点。

```dart
Offset _rectEdge(Offset rectCenter, Offset toward) {
  final dx = toward.dx - rectCenter.dx;
  final dy = toward.dy - rectCenter.dy;

  final hw = nodeW / 2;  // 矩形半宽
  final hh = nodeH / 2;  // 矩形半高

  // 分别计算射线到达左右边和上下边所需的参数 t
  double tx = dx != 0 ? hw / dx.abs() : double.infinity;
  double ty = dy != 0 ? hh / dy.abs() : double.infinity;

  // 取较小的 t，即先碰到哪条边就在哪里截断
  final t = min(tx, ty);

  return Offset(rectCenter.dx + dx * t, rectCenter.dy + dy * t);
}
```

**原理**：把矩形边看作参数方程 `P = center + t * direction`，分别求 x 方向和 y 方向到达边界的 t 值，取最小值即为射线最先碰到的边。

对于正向（水平/垂直）连线，`t` 由单轴决定，线是水平或垂直的。  
对于斜向（45°）连线，`tx` 和 `ty` 的大小取决于矩形的宽高比：
- 若 `nodeW > nodeH`（宽矩形），斜向线先碰到上下边，从上下边出发
- 若 `nodeW < nodeH`（高矩形），斜向线先碰到左右边，从左右边出发

当前 `nodeW=90, nodeH=44`，宽矩形，斜向线从上下边出发，视觉上自然。

### 3.4 点击命中检测

`CustomPainter` 本身不处理手势，通过 `GestureDetector` + `GlobalKey` 实现：

```dart
GestureDetector(
  onTapUp: (details) {
    // 1. 通过 GlobalKey 获取 CustomPaint 的 RenderBox
    final paintBox = _mapPaintKey.currentContext?.findRenderObject() as RenderBox?;

    // 2. 将全局坐标转换为 CustomPaint 内的本地坐标
    final localPos = paintBox.globalToLocal(details.globalPosition);

    // 3. 遍历所有出口，判断点击是否落在节点矩形内（扩大8px容错）
    for (final exit in exits) {
      final nodeCenter = _nodeCenter(exit.dir, center);
      final rect = Rect.fromCenter(
        center: nodeCenter,
        width: ExitMapPainter.nodeW + 16,   // 比渲染区域略大，方便触控
        height: ExitMapPainter.nodeH + 16,
      );
      if (rect.contains(localPos)) {
        _move(exit.dir);
        return;
      }
    }
  },
)
```

`_nodeCenter` 在 State 类和 Painter 类中使用完全相同的坐标计算逻辑，保证点击区域和渲染位置严格对应。

## 四、方向系统

支持8个方向，分为正向和斜向：

| dir | 中文 | 坐标偏移 | 类型 |
|-----|------|----------|------|
| n   | 北   | (0, -step)       | 正向 |
| s   | 南   | (0, +step)       | 正向 |
| e   | 东   | (+step, 0)       | 正向 |
| w   | 西   | (-step, 0)       | 正向 |
| ne  | 东北 | (+step, -step)   | 斜向 |
| nw  | 西北 | (-step, -step)   | 斜向 |
| se  | 东南 | (+step, +step)   | 斜向 |
| sw  | 西南 | (-step, +step)   | 斜向 |

斜向节点中心距当前节点中心为 `step * √2 ≈ 155.6px`，连线视觉长度比正向长约41%，这是正常的几何结果，符合直觉（斜路比直路远）。

## 五、视觉设计规范

### 颜色

| 元素 | 颜色 | 说明 |
|------|------|------|
| 背景 | `#0D0D0D` | 近黑，突出节点 |
| 普通节点边框 | `#B8860B`（暗金） | 未激活状态 |
| 当前节点边框 | `#FFD700`（亮金） | 高亮当前位置 |
| 当前节点文字 | `#FF3333`（红） | 仿古风格，醒目 |
| 普通节点文字 | `#FFFFFF`（白） | 清晰可读 |
| 普通连线 | `#B8860B`（暗金） | 与边框统一 |
| 跨区域节点边框 | `#FF8C00`（橙） | 区分跨区域 |
| 跨区域连线 | `#FF8C00`（橙） | 与节点统一 |
| 跨区域文字 | `#FFD700`（黄） | 醒目提示 |

### 节点尺寸

```
nodeW = 90px   节点宽度
nodeH = 44px   节点高度
step  = 110px  节点间距（中心到中心，正向）
```

间距 `step=110` 略大于节点宽度 `nodeW=90`，正向节点之间有 `110-90=20px` 的间隙，连线在间隙中可见。

## 六、Mock 数据系统

### 设计目的

服务端尚未支持斜向方向时，通过本地 mock 数据验证 UI 渲染效果，无需修改任何业务逻辑。

### 开关方式

```dart
// lib/services/game_api_service.dart
static const bool useMock = true;   // 改为 false 切回真实服务器
```

### Mock 数据结构

`MockData` 维护两张静态表：

```dart
// 房间出口表：roomId → List<Exit>
static final Map<String, List<Map<String, dynamic>>> _roomExits = { ... };

// 房间信息表：roomId → 区域/名称/类型
static final Map<String, Map<String, dynamic>> _roomInfo = { ... };

// 当前玩家位置（移动时更新）
static String currentRoom = 'room_misty_pass';
```

`mockMove` 在移动时更新 `currentRoom`，模拟服务端的状态持久化：

```dart
static PositionData mockMove(String currentRoomId, String direction) {
  final exit = _roomExits[currentRoomId]
      ?.firstWhere((e) => e['dir'] == direction, orElse: () => throw ...);
  currentRoom = exit['to_room'];   // 更新状态
  return mockPosition(currentRoom);
}
```

## 七、数据流

```
用户点击节点/按钮
       ↓
_move(direction)
       ↓
GameApiService.move()
  ├── useMock=true → MockData.mockMove() → 更新 currentRoom → 返回 PositionData
  └── useMock=false → HTTP POST /api/test/move/:id → 解析 JSON → 返回 PositionData
       ↓
setState(() => _positionData = data)
       ↓
_buildCrossMap() 重新构建
       ↓
ExitMapPainter.paint() 重绘画布
```

整个流程是单向数据流：服务端（或 mock）是唯一的状态来源，UI 完全由 `PositionData` 驱动，不维护任何本地地图状态。

## 八、扩展说明

### 添加新方向（如 u/d 上下）

1. 在 `_nodeCenter` 中添加坐标映射
2. 在 `ExitMapPainter._nc` 中添加相同映射
3. `_buildCrossMap` 的方向过滤列表中加入新方向
4. `_buildMoveButtons` 中添加对应按钮

### 调整地图布局密度

修改 `ExitMapPainter` 的常量：

```dart
static const double nodeW = 90.0;   // 节点宽，增大则节点更宽
static const double nodeH = 44.0;   // 节点高
static const double step = 110.0;   // 间距，增大则节点间距更大
```

`SizedBox(height: 280)` 是地图画布高度，若 `step` 增大需同步调整，确保边缘节点不被裁剪（最小安全高度 = `step * 2 + nodeH`）。
